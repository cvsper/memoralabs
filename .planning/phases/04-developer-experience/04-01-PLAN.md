Create a focused test file that verifies all error response paths return structured JSON. Test these scenarios:

1. **401 Unauthorized** - Call `POST /v1/memory` with no Authorization header. Assert response is `{"error": "UNAUTHORIZED", "message": "..."}`.
2. **401 Invalid key** - Call `POST /v1/memory` with `Authorization: Bearer ml_invalid`. Assert same structure.
3. **404 Not Found** - Call `GET /v1/memory/nonexistent-id` with valid auth. Assert `{"error": "NOT_FOUND", "message": "Memory not found"}`.
4. **422 Validation Error** - Call `POST /v1/memory` with valid auth but empty body `{}`. Assert `{"error": "VALIDATION_ERROR", "message": "Request validation failed", "details": [...]}` where details is a list.
5. **409 Conflict** - Call `POST /v1/auth/signup` twice with the same email. Assert `{"error": "CONFLICT", "message": "..."}`.
6. **No HTML in any error** - For each error response above, assert `Content-Type` header contains `application/json`.

Use the existing test patterns: `httpx.ASGITransport` + `httpx.AsyncClient`, `pytest.mark.asyncio`, the `tmp_path` fixture approach from existing tests. Set `DATA_DIR` and `VECTOR_INDEX_DIR` via monkeypatch on `app.config` and `app.main` modules (per project decision).

Each test function should assert:
- `response.status_code` matches expected
- `response.json()` has "error" key (string)
- `response.json()` has "message" key (string)
- Content-Type is application/json
- For 422: `response.json()` has "details" key (list)
